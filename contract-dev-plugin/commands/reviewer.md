## 当前分支信息
- 当前分支：!`git branch --show-current`
- 未提交改动（工作区 vs HEAD）：!`git status --porcelain`
- 与 master 分叉点：!`git merge-base master HEAD`
- 分支全量 diff（分叉点至 HEAD，已提交部分）：!`git diff $(git merge-base master HEAD) HEAD`
- 未提交改动 diff（工作区未 commit 部分）：!`git diff HEAD`
- openspec 活跃变更：!`ls openspec/changes/ 2>/dev/null | grep -v archive || echo "无"`

## 任务

通过 **Agent Team 并行** 执行三项审查任务：

---

### 阶段一：代码审查（Agent Team 并行）

启动 5 个并行 agent 从不同维度审查代码：

**Agent-1: 并发安全审查**
```
你是并发安全审查专家。请审查以下代码中的并发安全问题：
- 共享变量访问是否加锁
- 线程池配置是否合理
- 是否有竞态条件

diff：[传入 diff 内容]
```

**Agent-2: 事务边界审查**
```
你是事务边界审查专家。请审查以下代码中的事务问题：
- 事务传播级别是否正确
- 是否有长事务风险
- 事务内是否调用外部服务

diff：[传入 diff 内容]
```

**Agent-3: 空指针防护审查**
```
你是空指针防护审查专家。请审查以下代码中的空指针风险：
- 方法返回值是否可能为 null
- Optional 使用是否规范
- 集合遍历前是否检查空

diff：[传入 diff 内容]
```

**Agent-4: 幂等性审查**
```
你是幂等性审查专家。请审查以下代码中的幂等问题：
- 接口是否有幂等保护
- 重复提交如何处理
- 消息消费是否幂等

diff：[传入 diff 内容]
```

**Agent-5: 异常处理审查**
```
你是异常处理审查专家。请审查以下代码中的异常处理：
- 异常是否正确抛出
- 事务内异常是否导致回滚
- 是否有吞异常的情况

diff：[传入 diff 内容]
```

---

### 阶段二：测试代码审查（Agent Team 并行）

**如果本次变更包含测试代码，必须执行此阶段。**

启动 2 个并行 agent 审查测试代码：

**Agent-6: 测试代码一致性审查**
```
你是测试代码一致性审查专家。请审查测试代码与被测试代码之间的一致性：

审查要点：
1. 测试用例是否覆盖被测试代码的所有关键路径
2. 测试输入是否与实际业务场景匹配
3. 断言是否验证了正确的返回值和系统行为
4. Mock 行为是否与真实依赖一致
5. 测试数据是否与生产环境数据格式一致

diff：[传入 diff 内容]
```

**Agent-7: 测试代码有效性审查**
```
你是测试代码有效性审查专家。请审查测试代码的有效性和精简性：

审查要点：
1. 测试场景是否清晰，研发人员能否快速理解验证目的
2. 是否存在冗余或重复的测试用例
3. 测试是否验证系统行为的准确性（而不仅仅是代码覆盖率）
4. 边界条件和异常场景是否充分覆盖
5. 测试代码是否过于复杂，难以维护

diff：[传入 diff 内容]
```

---

### 阶段三：OpenSpec 文档一致性审查（Agent Team 并行）

**无论 openspec 目录是否有活跃变更，此阶段都必须执行。**

**Agent-8: 代码阅读者**
```
你是代码阅读者。请根据以下 diff 内容，读取本次变更涉及的源码文件，
整理出实际实现的行为，包括：工具/方法的参数名和类型、返回数据结构、核心业务规则、枚举值定义。
只做阅读和整理，不修改任何文件。输出格式：结构化的"代码行为摘要"。

diff 内容：[传入 diff 内容]
```

**Agent-9: 文档审查者**
```
你是文档审查者。请读取 openspec/changes/ 下所有非 archive 的变更目录中的文档
（包括 design.md、tasks.md、specs/ 下的规格文件），以及 openspec/specs/ 下的主规格文件。
整理出文档描述的行为，包括：接口参数、返回结构、业务规则、触发条件。
同时检查：tasks.md 中是否有未完成的 - [ ] 任务，是否存在"待定"、"TODO"等模糊描述。
只做阅读和整理，不修改任何文件。输出格式：结构化的"文档描述摘要"。
```
