# 独立审查阶段（Evaluator）

## 核心理念

**自评不可信，必须切换角色。**

代码写完后，不要说"再检查一下有没有问题"，而是切换成明确的 reviewer 角色，从以下维度挑刺：

> 现在你不是作者，你是资深 reviewer。请从并发、事务边界、空指针、幂等、异常回滚、日志、监控、兼容性、测试遗漏几个方面挑问题。先只列问题，不要立刻修。

## 审查维度

### 1. 并发安全
- 共享变量访问是否加锁
- 线程池配置是否合理
- 是否有竞态条件

### 2. 事务边界
- 事务传播级别是否正确
- 是否有长事务风险
- 事务内是否调用外部服务

### 3. 空指针防护
- 方法返回值是否可能为 null
- 集合遍历前是否检查空
- Optional 使用是否规范

### 4. 幂等性
- 接口是否有幂等保护
- 重复提交如何处理
- 消息消费是否幂等

### 5. 异常回滚
- 异常是否正确抛出
- 事务内异常是否导致回滚
- 是否有吞异常的情况

### 6. 日志规范
- 关键操作是否有日志
- 敏感信息是否脱敏
- 日志级别是否合理

### 7. 监控告警
- 核心指标是否上报
- 异常是否告警
- 是否有埋点

### 8. 兼容性
- 接口变更是否兼容旧版本
- 配置变更是否平滑
- 数据库 schema 变更是否兼容

### 9. 测试覆盖
- 核心逻辑是否有测试
- 边界条件是否覆盖
- 是否有遗漏的测试场景

## 审查流程

1. **切换角色**：明确自己现在是 reviewer，不是作者
2. **逐维检查**：按上述 9 个维度逐一审查
3. **只列问题**：输出问题清单，不立刻修复
4. **用户决策**：询问用户是先修问题还是继续

## 输出格式

```markdown
## 审查报告

### 并发安全
- [问题描述] - 位置

### 事务边界
- [问题描述] - 位置

...
```
