# 测试开发规范

## 概述
销售项目测试代码相关规范，包括 TDD 原则和实践技巧。

## 测试代码路径
- 路径：`utopia-nrs-sales-project-start/src/test/java/com/ke/utopia/nrs/salesproject`
- 命名：保证清晰易读的命名和分组

---

## TDD 工作流

### 循环

```
🔴 RED → 编写失败的测试
    ↓
🟢 GREEN → 编写最少代码使测试通过
    ↓
🔵 REFACTOR → 改进代码质量
    ↓
   重复...
```

### 三定律

1. 只为使失败的测试通过而编写生产代码
2. 只编写刚好能证明失败的测试
3. 只编写刚好能使测试通过的代码

### RED 阶段

**编写内容**：
| 关注点 | 示例 |
|-------|---------|
| 行为 | "应当将两个数字相加" |
| 边界情况 | "应当处理空输入" |
| 错误状态 | "应当对无效数据抛出异常" |

**规则**：
- 测试必须首先失败
- 测试名称描述预期行为
- 每个测试一个断言（理想情况）

### GREEN 阶段

**原则**：
| 原则 | 含义 |
|-----------|---------|
| **YAGNI** | 你不会需要它（You Aren't Gonna Need It） |
| **最简单的事情** | 编写最少代码使测试通过 |
| **不优化** | 只要能工作就行 |

### REFACTOR 阶段

**改进内容**：
| 领域 | 操作 |
|------|--------|
| 重复 | 提取公共代码 |
| 命名 | 明确意图 |
| 结构 | 改进组织 |
| 复杂度 | 简化逻辑 |

---

## AAA 模式

每个测试遵循：

| 步骤 | 目的 |
|------|---------|
| **Arrange（准备）** | 设置测试数据 |
| **Act（执行）** | 执行被测代码 |
| **Assert（断言）** | 验证预期结果 |

---

## 测试覆盖矩阵

### 何时使用
当测试方法较多、场景复杂时，用测试覆盖矩阵让研发快速了解测试覆盖情况。

### 写法示例

```java
/**
 * queryPersonalContractByCompositeOrderNos 场景测试
 *
 * <p>
 * 测试覆盖矩阵：
 * <pre>
 * +---------------------------+---------------+-----------+---------------+------------+----------+--------------------------------------------------------+
 * | 场景                      | S单关联合同    | 合同存在   | 合同类型=销售 | 签约人存在 | 预期结果 | 测试方法                                               |
 * +---------------------------+---------------+-----------+---------------+------------+----------+--------------------------------------------------------+
 * | 入参为空列表               | -             | -         | -             | -          | 空列表   | {@link #testEmptyInput_ReturnEmptyList()}               |
 * | 入参为null                | -             | -         | -             | -          | 空列表   | {@link #testNullInput_ReturnEmptyList()}                |
 * | 成功返回                  | 是            | 是        | 是            | 是         | 返回DTO  | {@link #testSuccess_ReturnDTO()}                      |
 * +---------------------------+---------------+-----------+---------------+------------+----------+--------------------------------------------------------+
 * </pre>
 * </p>
 */
```

### 关键点
1. **矩阵列**：条件维度 + 预期结果 + 测试方法
2. **`{@link #methodName()}` 语法**：IDE 中可点击跳转
3. **方法命名**：`test场景_预期结果()` 格式

## 纯 Mock 测试

### 何时使用
当测试不依赖真实 Spring 容器、仅验证方法逻辑时，使用纯 Mock 测试。

### 优点
- **执行速度快**：从分钟级降到秒级
- **隔离性好**：不依赖外部服务

### 写法示例

```java
@RunWith(MockitoJUnitRunner.class)  // 不需要继承 BaseTest
@Slf4j
public class XxxTest {  // 不需要 extends BaseTest

    @InjectMocks
    private XxxService xxxService;  // 被测服务

    @Mock
    private XxxRpc xxxRpc;  // 依赖的 RPC/Service 用 @Mock

    @Test
    public void testXxx() {
        // 直接 mock 返回值，不需要启动容器
        when(xxxRpc.someMethod(any())).thenReturn(expectedResult);
        // 执行测试
        // 断言
    }
}
```

### 注意事项
- 不需要 `@Before` 初始化 `MockitoAnnotations`（`MockitoJUnitRunner` 自动处理）
- 所有依赖用 `@Mock` 注入
- 适合业务逻辑验证，不适合需要真实 DB/容器的测试

## 测试优先级

| 优先级 | 测试类型 |
|----------|-----------|
| 1 | 正常路径 |
| 2 | 错误情况 |
| 3 | 边界情况 |
| 4 | 性能 |

---

## 何时使用 TDD

| 场景 | TDD 价值 |
|----------|-----------|
| 新功能 | 高 |
| Bug 修复 | 高（先写测试） |
| 复杂逻辑 | 高 |
| 探索性开发 | 低（先探索，再 TDD） |
| UI 布局 | 低 |

---

## 反模式

| ❌ 不要做 | ✅ 应该做 |
|----------|-------|
| 跳过 RED 阶段 | 先看到测试失败 |
| 先写代码后写测试 | 先写测试后写代码 |
| 过度设计初始实现 | 保持简单 |
| 多个断言 | 每个测试一个行为 |
| 测试实现细节 | 测试行为 |

---

## AI 增强 TDD

### 多智能体模式

| 智能体 | 职责 |
|-------|------|
| Agent A | 编写失败的测试（RED） |
| Agent B | 实现使其通过（GREEN） |
| Agent C | 优化（REFACTOR） |

---

> **记住：** 测试即规范。如果你无法编写测试，说明你不理解需求。
